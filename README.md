## What Each Part Covers

### Part 1: Manufacturer Entity Resolution

**1a through 1f** examine the quality of the existing normalization pipeline:

- Raw vs. normalized distinct name counts (9,680 raw strings reduced to 510)
- How many raw names were actually changed (99.4%)
- What normalization fixes well (case, punctuation) and what it misses (typos, abbreviations, OCR errors)
- A critical finding: lossy normalization maps real manufacturer names to "Undefined" and vice versa, affecting a meaningful number of units
- An error taxonomy classifying the 510 normalized names into junk categories (undefined, generic words, model numbers, placeholders, OCR artifacts)

**Part 1 (cont.)** proposes an improved five-stage entity resolution pipeline and implements a lightweight proof of concept:

- Filters junk names before matching
- Blocks by first token to reduce O(n^2) comparisons
- Scores pairs using a combined token-sort and token-set ratio
- Uses union-find to cluster matches and selects a canonical name by unit frequency
- Includes validation examples showing correct merges and borderline cases at the threshold

---

### Part 2: Project Duplicate Detection

Identifies projects that represent the same real-world construction job uploaded multiple times.

**Approach:**
- Blocks within the same company by first four alpha characters of project name
- Scores five signals per pair: fuzzy name similarity (weight 0.45), engineer overlap (0.20), manufacturer fingerprint overlap (0.15), sheet count proximity (0.10), equipment count proximity (0.10)
- Splits results into auto-merge tier (score >= 0.80) and needs-review tier (0.65 to 0.80)
- Uses union-find to cluster pairs into duplicate groups

**Outputs:**
- Percentage of projects involved in duplicates
- Group size distribution
- Duplication rate by company and market category
- Side-by-side display of the largest duplicate groups for visual confirmation
- False-positive spot check from the needs-review tier
- Canonical selection logic and a field-level merge strategy table

**Note on unnamed projects:** 896 projects with empty names are excluded from name-based blocking to avoid false positives. A production system would run a separate pass on these using only engineer overlap, manufacturer fingerprint, sheet count, and created_at proximity signals.

### Part 3: Exploratory Analysis

Four visualizations targeting questions a manufacturer rep would care about, plus a competitive intelligence section using hidden schedule data.

**Viz 1: Market share**
Top 20 manufacturers by total units specified. Titus, Price, Greenheck, and Trane together account for roughly 42% of all non-undefined units.

**Viz 2: Category specialization**
Heatmap showing each manufacturer's unit share by equipment category. Greenheck dominates Fans, Titus leads Supply Diffusers, and Mitsubishi is the clear Split VRF specialist.

**Viz 3: Co-occurrence**
How often pairs of manufacturers appear on the same projects. Titus and Greenheck are the most common pairing, appearing together on thousands of jobs because they cover complementary categories. High co-occurrence between direct competitors reveals displacement opportunities.

**Viz 4: Time trends**
Monthly project upload volume alongside unit trends for the top 6 manufacturers. The dataset spans fewer than 12 months, so no seasonal conclusions are drawn.

**Competitor Brands in the Market (Hidden Schedule Analysis)**
Identifies manufacturers appearing heavily in hidden schedules but rarely in visible ones. These are competitor brands that reps encounter on jobs but do not sell, and represent targets for rep expansion or displacement analysis.
